## Bug Pattern

The bug pattern is assigning a resource pointer to a shared/persistent location (e.g., `test->priv = sb`) **before** subsequent initialization steps that may fail and return early. When initialization fails midway, the cleanup/exit routine still runs and dereferences the partially-initialized pointer — but because the pointer was stored prematurely, the cleanup code assumes the resource is fully set up. The fix involves either (1) deferring the assignment until after all initialization succeeds, and/or (2) checking whether the pointer is valid/non-NULL in the cleanup function before dereferencing it. The root cause is a mismatch between the lifecycle of a stored pointer and the actual initialization state of the resource it points to, leading to NULL pointer dereference (or use of uninitialized state) during cleanup after a failed init.
